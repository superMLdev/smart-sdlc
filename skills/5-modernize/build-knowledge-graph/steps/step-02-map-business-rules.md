## What Is a Business Rule?

A business rule is a constraint, calculation, or decision that the business requires to be true — independent of any technology.

| Not a business rule | A business rule |
|---------------------|-----------------|
| COBOL paragraph VALIDATE-CREDIT-CHECK | "An order cannot be placed if it would cause the customer's balance to exceed their credit limit" |
| DB2 CHECK constraint on STATUS_CD | "Order status must be one of: Pending, Confirmed, Shipped, Cancelled, Returned" |
| IF statement in ORDVALD | "Orders with total amount over $10,000 require manager approval before processing" |
| VSAM KSDS indexed file | "Each customer has exactly one master record identified by customer ID" |

The rule is the business intent. The code is just how the legacy system happened to enforce it.

## Business Rule Sources

Extract rules from multiple sources in the legacy inventory:

### Source 1: COBOL Conditional Logic

Review all `IF`, `EVALUATE`, and `PERFORM ... UNTIL` blocks flagged as business rules in programs.md.

For each, write one or more rules:

```
Source: ORDVALD.cbl, paragraph 0210-CHECK-CREDIT-LIMIT
Code:
  IF WS-ORDER-TOTAL + CUST-BALANCE > CUST-CREDIT-LIMIT
     MOVE 'Y' TO WS-OVER-LIMIT-FLAG
     
→ RULE-001: An order cannot be processed if (order total + current customer balance) exceeds the customer's credit limit.
  Type: Validation
  Entity: Order, Customer
  Trigger: Order placement
  Condition: order.total + customer.balance > customer.creditLimit
  Action: Reject order
  Source: ORDVALD.cbl:0210-CHECK-CREDIT-LIMIT
  Confidence: High
```

### Source 2: 88-Level Values and Reference Data

Every set of 88-level values defines a valid-value business rule:

```
Source: CUSTMSTR.cpy
Code:
  88 CUST-ACTIVE   VALUE 'A'
  88 CUST-SUSPENDED VALUE 'S'
  88 CUST-CLOSED   VALUE 'C'
  
→ RULE-002: Customer status must be one of: Active (A), Suspended (S), or Closed (C).
  Type: State constraint
  Entity: Customer
  Source: CUSTMSTR.cpy
  Confidence: High
```

### Source 3: DB2 / SQL Check Constraints

```
Source: ORDERS DDL
Code: CHECK (STATUS IN ('P','C','S','X','R'))

→ RULE-003: Order status must be one of: P=Pending, C=Confirmed, S=Shipped, X=Cancelled, R=Returned.
  Type: State constraint
  Entity: Order
  Source: ORDERS DDL
  Confidence: High (need to confirm status labels from domain expert)
```

### Source 4: COMPUTE / CALCULATE Statements

```
Source: ORDTOTAL.cbl, paragraph 0400-CALCULATE-TAX
Code:
  COMPUTE WS-TAX ROUNDED = WS-SUBTOTAL * WS-TAX-RATE / 100

→ RULE-004: Tax is calculated as: tax = subtotal × (tax_rate / 100), rounded.
  Type: Calculation
  Entity: Order
  Source: ORDTOTAL.cbl:0400-CALCULATE-TAX
  Note: Tax rate stored in WS-TAX-RATE — source of rate value needs investigation
  Confidence: High (formula clear) | Rate source: [NEEDS-EXPERT-REVIEW]
```

### Source 5: JCL Conditional Execution

```
Source: ORDPROC.JCL
Code:
  //STEP030 EXEC PGM=ORDPOST,COND=(0,NE,STEP020)
  (Step 030 skipped if Step 020 returned non-zero)
  
→ RULE-005: Order posting must not proceed if order validation fails.
  Type: Process constraint
  Source: ORDPROC.JCL
  Confidence: High
```

### Source 6: External Documents (if provided)

For each document the user supplied, extract rules as:
```

→ RULE-{n}: {normalized rule statement}
  Type: {from document, not inferred}
  Confidence: Authoritative (from specification)
```

## Business Rule Categories

Group rules by type:

| Category | Description | Example |
|----------|-------------|---------|
| Validation | Input must satisfy a condition | Order amount must be positive |
| State constraint | Allowed states and transitions | Order status must be one of... |
| Calculation | How a value is derived | Tax = subtotal × rate |
| Authorization | Who can do what | Orders over $10K need approval |
| Temporal | Time-based conditions | Policy expires 12 months after start |
| Cardinality | Counts and limits | Customer can have max 5 active orders |
| Threshold | Minimum/maximum values | Credit limit minimum $0 |
| Dependency | One rule requires another | Cannot post order without validation |

```markdown
### RULE-002 — Customer status values
**Statement:** Customer status must be one of: Active, Suspended, or Closed.
**Valid values:**
  - `A` = Active — standard operating state
  - `S` = Suspended — orders blocked
  - `C` = Closed — permanent, no transactions
**Entities:** Customer
**Source:** CUSTMSTR.cpy (88-level values)
**Confidence:** ⭐ High

---

### RULE-015 — Manager approval threshold [NEEDS-EXPERT-REVIEW]
**Statement:** [INFERRED] Orders with total exceeding a threshold require manager approval.
**Evidence:** Field WS-MGR-APPROVAL-FLAG exists in ORDERMST and is set in ORDAUTH.cbl.
**Threshold value:** Unknown — stored in configuration table not found in analysis
**Question:** What is the approval threshold? Is it $10,000 or different per customer type?
**Entities:** Order
**Source:** ORDAUTH.cbl + ORDERMST (field MGR-APPROVAL-FLG)
**Confidence:** ⚠️ Needs expert review

---
```
