## Why This Step Matters

In legacy systems, **data structures ARE the business model**. COBOL copybooks, RPG physical files, DB2 DDL, and Hibernate mapping files encode the company's understanding of its own domain. Field names, sizes, PIC clauses, and VALUE constraints are business rules expressed as data.

A COBOL field like:
```cobol
05 CUST-CREDIT-CLASS     PIC X(1).
   88 GOLD-CUSTOMER      VALUE 'G'.
   88 SILVER-CUSTOMER    VALUE 'S'.
   88 STANDARD-CUSTOMER  VALUE 'N'.
   88 SUSPENDED          VALUE 'X'.
```

Is not just a data definition — it is the complete taxonomy of customer types in this business. Losing it loses a business rule.

## A. COBOL Copybooks (`.cpy`)

Copybooks are the most important files to read in any COBOL system — they are the shared data dictionary.

For each copybook, extract:

```
Copybook: {CUSTOMER-RECORD}
Source file: {CUSTMSTR.cpy}
Used by programs: {CUSTENQ, CUSTUPD, ORDVALD, ...}  ← grep for COPY CUSTMSTR
Purpose: {Customer master record layout}

Fields:
  Level  Name                   Type        Length  Notes
  ─────  ─────────────────────  ──────────  ──────  ──────────────────────────────
  05     CUST-ID                PIC 9(8)    8 num   Customer identifier
  05     CUST-NAME              PIC X(40)   40 char Customer full name
  05     CUST-STATUS            PIC X(1)    1 char  {see 88-levels below}
    88   CUST-ACTIVE            VALUE 'A'           Customer is active
    88   CUST-SUSPENDED         VALUE 'S'           Customer suspended ⭐ RULE
    88   CUST-CLOSED            VALUE 'C'           Account closed ⭐ RULE
  05     CUST-CREDIT-LIMIT      PIC 9(7)V99 9.2 num Max credit allowed ⭐ RULE
  05     CUST-BALANCE           PIC S9(7)V99 9.2 num Signed — can be negative? [VERIFY]
  05     CUST-LAST-ORDER-DATE   PIC 9(8)    8 num   YYYYMMDD format [VERIFY]
```

Flag:
- `88` level items → explicit business states / classifications
- `V` in PIC clause → decimal placement (financial amounts)
- `S` in PIC clause → signed values (can go negative — business significance?)
- `REDEFINES` → same memory interpreted as different data types → complex business rule
- `OCCURS` → arrays of data → cardinality constraint

## B. RPG Physical Files (`.pf`)

Physical files are the database tables of AS400/iSeries:

```
Physical File: {CUSTP}
Source file: {CUSTP.pf}
Description: {from A*DESC spec line}

Fields:
  Name       Type    Length  Description         Notes
  ─────────  ──────  ──────  ──────────────────  ──────────────────
  CUSTID     P       8,0     Customer ID         Keyed field
  CUSTNAME   A       40      Customer Name
  CUSTSTATUS A       1       Status Code         'A'=Active, 'I'=Inactive [VERIFY]
  CREDITLIM  P       9,2     Credit Limit        ⭐ RULE
  ORDCOUNT   P       5,0     Order Count
  
Key fields: {CUSTID}
Unique key: ✅
```

Logical files (`.lf`) → extract as views — note the selection criteria (OMIT/SELECT WHEN) as they often encode business rules:
```
Logical File: {CUSTACTL}
Over: {CUSTP}
Key: {CUSTNAME}
Filter: SELECT WHEN CUSTSTATUS = 'A'  ← ⭐ RULE: Only active customers in this view
```

## C. DB2 / SQL Schema

Read DDL files or DCLGEN output:

```
Table: {CUSTOMER}
Source: {CUSTMSTR.sql | DCLGEN output}

Columns:
  Name              Type          Null?  Constraint / Default    Notes
  ────────────────  ────────────  ─────  ──────────────────────  ─────────────────
  CUST_ID           CHAR(8)       NOT NULL PRIMARY KEY
  CUST_STATUS       CHAR(1)       NOT NULL DEFAULT 'A'            ⭐ RULE: default active
  CREDIT_LIMIT      DECIMAL(9,2)  NOT NULL DEFAULT 0
  CREDIT_CLASS      CHAR(1)              CHECK IN ('G','S','N')  ⭐ RULE: enum constraint
  EFFECTIVE_DATE    DATE          NOT NULL
  EXPIRY_DATE       DATE                                          Nullable = no expiry?

Indexes: {list with uniqueness}
Foreign Keys: {list with ON DELETE behavior}
Check Constraints: {list — these are explicit business rules}
```

Flag: `CHECK` constraints → explicit business rules in code. Extract every one.

|-------|------|------------|-----------------|
| CUST-ID | PIC 9(8) | Required | Customer identifier |

| Code | Label | Meaning |

## E. Significant Value Tables (Reference Data)

Look for programs or tables that define code/value mappings:

```
Examples to search for:
  - Programs with names like: CODETAB, REFTAB, LOOKUP, TABLMGR
  - Tables named: CODE_MAST, REF_DATA, LOOKUP_TABLE
  - COBOL WORKING-STORAGE with large tables of 88-level values
  - EVALUATE WHEN blocks covering many cases
```

These are often the most business-rule-dense parts of the system.

⏸️ **STOP** — Present data dictionary and flag any fields needing domain expert clarification.

**Standard questions to ask:**
1. "Are there any fields with codes/values where you can explain the business meaning of each code?"
2. "Are there any REDEFINES structures where you can explain when each interpretation is used?"
3. "Are there date format fields where the valid range has business significance?"
